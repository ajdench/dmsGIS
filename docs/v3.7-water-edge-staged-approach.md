# v3.7 Water Edge Staged Approach

This note records the agreed staged approach for treating inlets, rivers, estuaries, and related Devolved Administrations border artifacts.

It is a design and execution note, not yet a statement that the geometry products have changed.

## Why This Is Needed

The current preprocessing chain clips exact boards to a hole-free landmask and then simplifies them, but it does not yet classify external water-facing edges.

That means the shipped family still lacks a formal distinction between:

- true sea coast
- estuary / inlet edges
- river or inland-water-facing edges
- internal administrative shared borders

That gap can surface as:

- overly deep inland coastal-looking projections
- visually noisy hydro-facing arcs
- Region-border artifacts across internal Devolved Administrations HB boundaries

## Staged Approach

### Stage 1. Formalize the tolerance and styling model

Create one shared versioned config for:

- preprocessing tolerances in projected metric units
- runtime visual tokens for inland-arc styling

This stage does not change geometry behavior yet.

It defines the controllable variables and makes them restart-safe.

Current status:

- the shared draft config now exists in:
  - `src/lib/config/waterEdgeTreatment.json`
- draft hydro-normalized landmask profile outputs are now materialized for review:
  - `geopackages/outputs/uk_landmask/UK_Landmask_OSM_simplified_v01_hydronormalized_coarse.geojson`
  - `geopackages/outputs/uk_landmask/UK_Landmask_OSM_simplified_v01_hydronormalized_standard.geojson`
  - `geopackages/outputs/uk_landmask/UK_Landmask_OSM_simplified_v01_hydronormalized_fine.geojson`
  - `geopackages/outputs/uk_landmask/UK_Landmask_OSM_simplified_v01_hydronormalized_profiles.json`
- the `standard` profile is now activated on the rebuild path as a review cutover
- that means shipped runtime boards rebuilt through `scripts/build-runtime-geometry-family.mjs` now clip against the `standard` hydro-normalized landmask profile
- the active hydro-normalized landmask is now class-aware:
  - only `inlandWater` deltas are folded into the clipping landmask
  - `estuary` deltas are still classified and published, but are not yet folded into the shipped clipping mask
- this is still a review state, not yet an accepted final baseline

### Stage 2. Add external-arc classification in preprocessing

Introduce a preprocessing seam that classifies edges or components into:

- `sea`
- `estuary`
- `inlandWater`
- `internal`

This should happen before downstream runtime fills, outlines, masks, and helper products are emitted.

Current status:

- review-time classified arc helper products are now materialized under:
  - `geopackages/outputs/water_edges/UK_ICB_LHB_v10_water_edge_classes.geojson`
  - `geopackages/outputs/water_edges/UK_Health_Board_2026_water_edge_classes.geojson`
  - `geopackages/outputs/water_edges/water_edge_classification_summary.json`
- published app-readable copies now also materialize under:
  - `public/data/regions/UK_ICB_LHB_v10_water_edge_classes.geojson`
  - `public/data/regions/UK_Health_Board_2026_water_edge_classes.geojson`
- these are heuristic review artifacts built from the active hydro-normalized profile
- they now drive the first live border-modifier treatment in runtime:
  - `inlandWater` arcs are folded into the real board-border and selected dissolved-Region border paths, not shown as a standalone blue review layer
  - those arcs now use a sea-colour mask plus a hard-opacity redraw so deep inland water-facing projections read as dimmed border, not full coastal stroke
  - the active hard-opacity is currently `50%` for both ordinary board-border treatment and selected dissolved-Region treatment
  - selected dissolved-Region treatment now uses preprocessed per-group inland-water subset outline files under `public/data/regions/outlines/*_inland_water.geojson` rather than resolving generic classified arcs at runtime
  - `estuary` arcs remain classified and published, but are not yet folded into the live border modifier path
  - this still does not yet replace the final shipped-family rebuild logic; it is the first live rendering cut of the classified result
- the classification schema is now intentionally broader than the first operational cut, so later refinement can add:
  - `harbourDock`
  - `canalCut`
  - `lakeReservoir`
  - `intertidal`
  - `islandFragment`

### Stage 3. Rebuild the shipped family from the classified result

Regenerate in one chain:

1. `Current` shipped fills
2. `2026` shipped fills
3. topology/shared-edge products
4. group/selection outline products
5. land/sea mask products
6. facility placement outputs

### Stage 4. Apply visual treatment in runtime

Runtime should not invent new geometry.

Runtime should only consume the classified family and style it via tokens, for example:

- inland arcs inherit Region-border colour
- inland arcs use hard opacity rather than fake coastal capping
- zoom can switch between prebuilt coarse/standard/fine products or visual profiles

## Recommended Token Split

### Preprocessing tokens

Use projected metric tolerances for:

- `minTrueCoastOpeningWidthM`
- `maxInlandWaterProjectionWidthM`
- `maxInlandProjectionDepthM`
- `minIslandAreaM2`
- `coastalFacilityInsetM`
- `devolvedInternalArcSuppressionWidthM`

These govern geometry truth and should be versioned with the product build.

### Runtime tokens

Use visual tokens for:

- `inlandArcColorMode`
- `inlandArcOpacity`
- `inlandArcSelectedOpacity`
- `inlandArcDash`
- zoom-band visual profile selection

These govern appearance only.

## Initial Guidance Source

The initial draft tolerances are heuristic and informed by general cartographic guidance, including:

- scale-dependent hydro portrayal
- more simplification at smaller scales
- less simplification at larger scales

These values are not yet treated as final truth.

## Important Red Line

Do not rely on runtime styling alone to fix hydro/coastal geometry problems.

The long-term fix is:

- classify in preprocessing
- rebuild the shipped family
- then style the resulting classes consistently in runtime

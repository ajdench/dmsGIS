# v3.5 Geometry Architecture Redlines

This note records the architectural conclusion reached after the repeated `Current` split-ICB, outline, landmask, and highlight regressions.

It is intentionally short, prescriptive, and restart-friendly.

## Problem Summary

The recurring regressions are not mainly runtime UI bugs.

They come from the fact that the geometry family is still not fully atomic in `Current`.

There are still too many partially independent seams between:

1. exact canonical parent boards
2. exact ward-level split assignment
3. parent-edge remainder repair
4. dissolved shipped split runtime
5. selected Region outline products
6. `Land` / `Sea` mask products
7. facility inward snapping against shipped polygons

`2026` is much cleaner because it does not carry the same split-parent workflow.

## Required Architecture

For both `Current` and `2026`, one geometry family must exist:

1. exact canonical board source
2. shipped runtime fill polygons
3. authoritative shared-edge / outline products derived from those shipped fills
4. `Land` / `Sea` mask products derived from those shipped fills
5. facility preprocessing validated against those shipped fills

For `Current`, the split-ICB workflow must be inside that same family, not a side seam:

1. canonical parent outer boundary from the exact `Current` board source
2. ward-level internal assignment from official ward `BFC`
3. contiguity cleanup at ward-component level before dissolve
4. dissolved shipped split runtime clipped exactly to the shipped parent
5. outlines and `Land` / `Sea` masks built only after zero-drift validation

## Red Lines

Do not:

- patch runtime styling to hide bad geometry
- rebuild masks from anything except the final shipped runtime board family
- generate selected-outline products through mixed mesh/exterior logic for split-aware `Current` groups
- allow a split runtime feature to extend outside its shipped parent
- allow selected split-Region outlines to include internal split arcs
- rely on manual visual spot-fixing instead of geometry assertions

## Required Invariants

Across both `Current` and `2026`:

- shipped fills are hole-free
- shared borders are continuous
- neighboring fills do not overlap materially
- masks contain shipped coast truth where intended
- facilities sit inside assigned shipped board products
- outlines are derived from the shipped family, not a parallel source

Additional `Current` split invariants:

- ward assignments are contiguous by intended Region body
- orphan ward components are either justified islands or reassigned
- Winchester / Worthy Down style orphaning is not allowed
- selected Region outline follows ward-assigned extent only
- split parent coast and landmask remain exact to the shipped result

## Sidebar / Runtime Contract

Do not redesign the app shell for this work.

Preserve:

- `viewPresets.json`
- `boundarySystems.ts`
- existing store contracts
- Overlay / Region / Basemap sidebar behavior
- selection and highlight behavior
- Playground dynamic Region behavior

The rebuild should happen underneath the current app contract, not via a sidebar/runtime rewrite.

## Immediate Execution Direction

The next proper geometry pass should:

1. freeze current app behavior as acceptance criteria
2. rebuild the `Current` split exact workflow with explicit ward-component contiguity cleanup
3. rebuild shipped `Current` runtime fills from that corrected split source
4. rebuild `Current` outlines and `Land` / `Sea` masks from those shipped fills
5. re-run facility inward preprocessing against the rebuilt shipped `Current` polygons
6. run the same product-family invariants across `2026`, even if little changes there

## Practical Interpretation

`2026` already behaves much closer to the desired model.

`Current` is the special case that still needs to be made truly atomic.

The goal is not “more fixes.”

The goal is to make it difficult for preprocessing to go backwards at all.

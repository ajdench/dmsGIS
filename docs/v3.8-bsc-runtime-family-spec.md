# v3.8 BSC Runtime Family Specification

This document defines the next full geometry rebuild direction after the rejected `v3.7` coastal-envelope compare work.

The new direction is:

- stop hybrid external-envelope reconstruction
- stop runtime/helper-line compensation for inland projections
- rebuild the shipped runtime board families directly from official `BSC` admin products wherever available
- preserve the app contract, ids, store/config behavior, and interaction semantics

This is a preprocessing specification, not a runtime redesign.

For the concrete restart/rebuild execution brief with renamed internal stage
outputs and explicit stable public contracts, also read:

- `docs/paired-current-2026-product-success-path.md`

## Paired Build Contract

`Current` and `2026` are not separate unrelated rebuilds.
They are one paired family build that branches once.

The accepted order is:

1. start from one shared canonical board/coastal truth
2. build `Current` as the unmerged board family
3. build `2026` by applying the NHS 2026 merge/redraw cases to that same family
4. apply the same runtime preparation philosophy to both branches
5. derive scenarios, masks, outlines, topology edges, and facilities afterward

This means:

- England ICB `BSC` remains the canonical England coastal / board truth
- ward `BSC` is only for the three special `Current` split internals
- `2026` mergers happen before runtime export, not after
- internal borders must remain contiguous through the merge stage itself

The active paired-build measurement artifact is:

- `geopackages/outputs/v38_bsc_runtime_family/paired_current_2026_alignment_report.json`

That report measures the seven changed England 2026 boards against unions of their
`Current` predecessor boards at both:

- source stage
- runtime stage

Use that report when investigating whether `2026` has drifted away from the `Current`
visual baseline on shared coast.

## Decision

The next version should rebuild both:

- `Current`
- `2026`

from a `BSC-first` runtime source family.

The objective is not to preserve the finest internal line detail.
The objective is to:

- materially simplify coastlines and inland water projections
- reduce shipped geometry complexity and file size
- keep the land-sea mask workflow that is already working
- preserve all app-facing contracts and behaviors

## Why This Replaces The `v3.7` Compare Path

The rejected `v3.7` compare work proved that these paths are not the right main solution:

- runtime/helper overlay dimming of inland-water arcs
- clipping old canonical boards to new envelopes
- hybrid reconstruction from old internal networks plus new external shells
- region-envelope compare products such as `bfe` / partial East-only prototypes

Those approaches introduced too many reconstruction seams and review-only artefacts.

The accepted simplification is now:

- use cleaner official `BSC` products directly as the shipped runtime board family
- keep canonical exact products and exact split sources for provenance, facility processing, and rebuild discipline
- let the runtime family be deliberately coarser

## Core Architectural Rule

There are two geometry families and they must not be conflated:

1. Exact canonical family
- detailed official / official-equivalent exact products
- used for provenance, audits, facility processing, and rebuild inputs

2. Shipped runtime family
- official `BSC`-first runtime board products
- used by the app for fills, boundaries, topology edges, outlines, masks, and visible alignment

The runtime family is the operational truth for the browser.
The exact family remains the source/provenance truth upstream.

## Scope

This specification applies to:

- `Current`
- `2026`
- board products
- split-ICB products
- shared-edge products
- dissolved Region outlines

- facility containment and coastal snap checks

It does not authorize:

- sidebar redesign

- preset name/code changes
- runtime selection logic redesign
- Playground contract redesign

The following must remain unchanged unless a later explicit product decision says otherwise:

- `boundary_code` identities
- `boundary_name` identities
- preset ids and labels
- `viewPresets.json` grouping semantics
- overlay family ids and meanings
- `Current` vs `2026` boundary-system split
- point selection behavior
- boundary selection behavior
- dissolved Region highlight semantics
- sidebar/store/popover behavior
- GitHub Pages static-delivery model

Operationally, this means the app should keep consuming the same kinds of files at the same public paths, even if the rebuilt geometry inside those files becomes coarser.

- `2026` runtime boards now rebuild from the mixed `v3.8` source family
  - `29` direct official England ICB `BSC`
  - `4` merge-only England boards now keep exact SICBL-derived 2026 internals but are constrained to predecessor `Current` / official ICB `BSC` shells
  - `3` Frimley-driven England boards now keep exact 2026 redraw logic but are constrained to those same predecessor `Current` / official ICB `BSC` shells
  - `26` upstream devolved boards rebuilt through one gentle shared `BSC-like` simplification path

One important implementation note from this first cut:

- `scripts/build_current_split_icb_runtime.py` had to be hardened so the split runtime export clips to live parent geometry in projected CRS before final reprojection, then normalizes again after reprojection
- without that, `E54000025` leaked tiny invalid/overlapping slivers back outside the new `Current` `BSC` parent family and failed the geometry validation gate

One important 2026 correction from the first live cut:

- the `4` merge-only England 2026 boards must not be rebuilt as whole-ICB `BSC` dissolves
- that wrongly reused complete current ICB polygons where the true 2026 change is a SICBL-based recombination
- the result was large runtime overlaps and visible darker double-fill in East England
- the corrected source rule is:
  - keep `29` unchanged England boards on direct official ICB `BSC`
  - keep the `4` merge-only England boards on exact SICBL-derived 2026 internals, but rebuild their overlapping East component inside predecessor `Current` / official ICB `BSC` shells
  - keep the `3` Frimley-driven England boards on exact 2026 redraw geometry, but rebuild that overlapping component inside those same predecessor `Current` / official ICB `BSC` shells
- a later improvement can still replace those `7` changed boards with a dedicated `BSC-like` redraw, but not with whole-ICB dissolves

One further shipped runtime correction now also applies:

- after `2026` runtime simplification, those same changed-board components are rebuilt again from the already-built `Current` runtime family as the shell baseline
- at runtime they also yield to already-built neighboring unchanged 2026 boards where those neighbors meet them, so the final `2026` family stays overlap-free
- this prevents the shipped runtime file from drifting back outside the shared `Current` coastal envelopes after simplify without reopening board overlaps against unchanged 2026 neighbors
- the active paired-stage measurement artifact is:
  - `geopackages/outputs/v38_bsc_runtime_family/paired_current_2026_alignment_report.json`
- current measured result there:
  - source-stage max extra area outside predecessor `Current` shells: `0.0 m²`
  - runtime-stage max extra area outside predecessor `Current` shells: `333.9 m²`

## `Current` Build Specification

### Base runtime boards

Rebuild the `Current` shipped runtime board family from `BSC-first` sources.

For England:

- shipped board polygons should come directly from official England ICB `BSC`

For devolved administrations:

- shipped runtime polygons should come from direct `BSC`-grade sources where available
- otherwise from a reproducible documented generalisation pass from the best official-equivalent exact source

### `Current` split-ICB special cases

The three split-ICB special cases remain special and must continue to exist:

- `E54000025`
- `E54000042`
- `E54000048`

For these:

1. Parent runtime geometry must now come from the new `Current` runtime `BSC` parent family.
2. Internal split assignment still comes from the official ward exact path, not from runtime inference.
3. Split runtime products must be rebuilt against the new parent runtime geometry.
4. The resulting split runtime products must be clipped/validated against those new `BSC` parents.
5. The split runtime export may remain unsimplified if needed to avoid reintroducing overlap artefacts.

In short:

- parent shape = new `BSC` runtime parent
- internal assignment = exact ward source
- app product = clean split runtime product compatible with the `BSC` parent family

### Canonical wards source

## `2026` Build Specification

`2026` should move to the same runtime-family philosophy:

- shipped runtime `2026` boards should be `BSC-first`
- exact/provenance 2026 boards remain on the current exact workflow

For England `2026`:

- changed merger targets remain exact upstream
- shipped runtime delivery should still be rebuilt as the new `BSC`-first family

For devolved boards:

- same rule as `Current`: direct official `BSC` where available, otherwise documented `BSC-like` runtime derivation

Current implementation state:

- `2026` is now promoted to the `v3.8` runtime family
- the remaining source-composition caveat is explicit rather than blocking:
  - `4` merge-only England boards are rebuilt from exact SICBL-derived 2026 internals constrained to predecessor shells
  - `3` Frimley-driven England boards are rebuilt from exact 2026 redraw geometry constrained to predecessor shells, not direct official BSC
  - devolved runtime boards are still exact upstream pending any later optional `BSC-like` redraw decision

## Land-Sea Mask Specification

Keep the land-sea mask workflow.

The new rule is:

1. build the shipped runtime board family first
2. union that shipped runtime family
3. derive the runtime land mask from that union
4. derive the matching sea patch from that same runtime family

Do not try to use land/sea masks to repair board geometry.
The masks follow the shipped board family, not the other way around.

## Facility Alignment Specification

Facilities must continue to be validated against the shipped runtime family.

Rules:

- keep the baseline facilities file schema and app contract unchanged
- rerun containment against the rebuilt `Current` runtime boards
- rerun containment against the rebuilt `2026` runtime boards
- preserve reviewed coastal snaps where still valid
- re-evaluate any facility that now falls outside its assigned runtime board after the `BSC` cutover

The facility file can remain one stable public product; it does not need compare-token logic in the final promoted version.

## Derived Product Specification

After the runtime board family is rebuilt, regenerate from that same family:

- topology/shared-edge products
- scenario board derivatives
  - `Current` should stay on the rebuilt unmerged/current `BSC` board family
  - `SJC JMC`, `COA 3a`, `COA 3b`, and the COA Playgrounds should derive from the rebuilt merged `2026 BSC` board family
  - scenario products may still carry different assignment labels (`jmc_name`, `region_name`, `scenario_region_id`) and different colouring contracts on top of that merged basis
- dissolved Region outlines
- split internal arc helper product
- land/sea masks
- facility boundary assignments and validations

No derived product should be generated from stale pre-`BSC` runtime geometry.

## Validation Requirements

The validation gate must stay in force.

At minimum, assert:

- expected board counts for `Current` and `2026`
- expected split-parent counts and split runtime presence
- no hole regressions in shipped board products
- no split runtime overlap against sibling runtime boards beyond tiny tolerance
- masks contain the rebuilt shipped runtime family
- facilities remain contained or explicitly reviewed/snapped
- dissolved Region outlines do not leak split/internal admin arcs unexpectedly

## Implementation Sequence

1. Revert runtime token state back to the stable full-UK baseline before new rebuild work begins.
2. Acquire/confirm all required `BSC` runtime sources.
3. Build a full-UK `Current` `BSC-first` runtime family.
4. Rebuild `Current` split-ICB runtime against those new `BSC` parents.
5. Rebuild `Current` topology edges, outlines, masks, and facility checks.
6. Build a full-UK `2026` `BSC-first` runtime family.
7. Rebuild `2026` edges, outlines, masks, and facility checks.
8. Validate the whole family.
9. Only then promote into the default live runtime files.

## Red Lines

- Do not continue the East-only prototype path as the main implementation branch.
- Do not continue hybrid external-envelope clipping/reconstruction as the main solution.
- Do not rely on runtime helper lines to compensate for wrong coastal geometry.
- Do not change the app contract while changing the geometry family.
- Do not weaken the geometry validation gate.

## Acceptance Standard

This next version is successful if:

- the app contract is unchanged
- the full-UK runtime board family is visibly cleaner and simpler
- inland river/coastal projection artefacts are materially reduced by source choice rather than patched visually
- file sizes remain suitable for GitHub Pages static delivery
- facilities remain operationally correct
- the build is restart-safe and reproducible from documented sources

# v3.5 Full Geometry Redress

This note records the agreed `v3.5` direction after the incremental `v3.1` to `v3.4` geometry and basemap passes.

`v3.4` remains the recoverable rollback baseline.

`v3.5` is not another local repair pass. It is the first full coherent rebuild path for the live geometry products.

Implementation status:

- the first live `v3.5` rebuild has now been executed
- the active recoverable baseline is documented in `docs/current-app-baseline-v3.5.md`
- this note remains the architectural rationale for that rebuild path

## Why `v3.5` Exists

The incremental path improved several important things:

- canonical board provenance
- preprocessing coastline truth
- visible UK land/sea alignment
- coastal facility preprocessing
- hidden basemap coastline-stroke cleanup

But it also showed the limit of seam-by-seam repair:

- internal ICB/HB border continuity is still brittle
- shipped simplified board products still contain real interior holes
- visible land/sea conformance is still being judged against downstream derived products rather than one authoritative product family
- the internal border products are still extracted and repaired after polygon simplification instead of being first-class geometry outputs

So the next path should not continue patching those seams independently.

## `v3.5` Goal

Produce one coherent geometry family for the live app while preserving existing app behavior.

That means:

1. new app-facing polygon products
2. new authoritative shared-edge products
3. new visible `Land` / `Sea` masks
4. facility preprocessing rerun against the new shipped `Current` polygons
5. runtime wiring kept behaviorally the same

The user-facing app should keep:

- current selection/highlight behavior
- current overlay controls and mixed-state behavior
- current scenario behavior
- current Playground behavior
- current split-ward `Current` behavior
- current coastal-facility treatment contract

## Product Stack

`v3.5` should use this stack:

1. Exact canonical board polygons
   - `Current`
   - `2026`

2. Shipped app-facing polygon products
   - derived coherently from the canonical polygons
   - these define the operational coastline for the app

3. Authoritative shared-edge products
   - built directly from the same polygon family
   - not extracted later from already-simplified runtime products as a repair step

4. Visible basemap alignment masks
   - `Land` derived from the shipped app-facing polygon coast truth
   - `Sea` derived as the complement of that same truth
   - Republic of Ireland included deliberately in the same visible land/sea system where needed for clean Northern Ireland context

5. Facility preprocessing
   - facilities snapped inward to their assigned shipped `Current` polygon when needed
   - original coordinates and snap metadata preserved

## Important Architecture Rule

For `v3.5`, stop treating these as separate geometry seams:

- polygon fills
- internal border lines
- visible basemap land
- visible basemap sea
- coastal facility placement

They should be regenerated as one coordinated family.

## What `v3.5` Is Not

`v3.5` is not:

- a visible basemap provider switch
- a dual-map runtime
- another edge-repair pass on top of current simplified products
- another incremental mask tweak without rebuilding the geometry family underneath

## Safest Execution Sequence

1. Preserve `v3.4` as the rollback point.
2. Freeze current runtime behavior as the acceptance contract.
3. Rebuild new shipped polygon products for `Current` and `2026`.
4. Build new authoritative shared-edge products from those rebuilt polygons.
5. Build new visible `Land` / `Sea` masks from the shipped polygon coast truth.
6. Rerun facility preprocessing against the new shipped `Current` polygons.
7. Swap runtime paths to the new products without changing UI semantics.
8. Validate behavior and visual parity before retiring old products.

## Effect On Coastal Alignment

Yes, the `Land` / `Sea` masks should be regenerated in `v3.5`.

That is expected and desired:

- the masks should follow the new shipped polygon coast truth
- the internal borders should come from the same geometry family
- coastal facility treatment should be rerun against that same family

This is the opposite of the previous incremental approach, where those surfaces drifted apart.

## Resume Rule

When resuming geometry work after `v3.4`, the default assumption should now be:

- preserve `v3.4`
- rebuild coherently in `v3.5`
- do not add another local repair unless it is strictly necessary to complete the full product rebuild path
